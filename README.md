# Демонстрационная версия приложения нап Go


## Описание
Репозиторий содержит демонстрационный микросервис на языке go для демонстрации структуры приложения, 
подходов к разработке, таких как Clean Architecture Layout, файлов развертывания Docker и docker-compose, 
обслуживания Makefile, внедрения зависимостей (uber-go/fx),
пакетов для работы с PostgreSQL, Redis, ClickHouse, Kafka, Cron, 
а так же правил написания кода (coding-style), golint и unit-тестов.


### Структура приложения
Структура приложения основана [golang-standards/project-layout](https://github.com/golang-standards/project-layout/tree/master)

Содержит _(нажмите для перехода в соответствующему каталогу)_:

- [`build`](build/) # файлы для локальной сборки в Docker
- [`cmd`](cmd/) # точки входа в приложение (запуск проекта, запуск миграций, консумеров очередей)
- [`deployments`](deployments/) # файлы для развертывания в kubernetes
- [`docs`](cmd/) # swagger документация проекта
- [`internal`](internal/) # код самого приложения
  - [`app`](internal/app/) # файлы настройки приложения: конфиги, обработчики ошибок, логгер и т.д.
  - [`common`](internal/common/)  # общие файлы: конфиги, метрики, middleware и т.д.
  - [`domain`](internal/domain/) # независимые друг от друга доменные зоны приложения
    - [`sk`](internal/domain/sk/) # домен СК (страховые компании) - пример
  - [`validator`](internal/validator/) # свои правила валидации структур
- [`migrations`](migrations/) # файлы миграций БД
- [`pkg`](pkg/) # общие пакеты
- [`tmp`](tmp/) # папка для временных файлов

Любая доменная зона имеет внутреннюю структуру:
```
domain
    sk                  # название доменной зоны  
        http                # для http-хендлеров
            admin               # для хендлеров которые взаимодействуют с фронтом (админкой); доступны только внутри данного микросервиса
            public              # для хендлеров доступных публично; доступны всем
            api                 # для общения микросервисов между собой; доступны другим микросервисам, не доступны в интернете
        model           # модели (структуры) сервиса
        service         # сервис, содержится бизнес-логика
        storage         # репозитории, для связи с базой данных
            model           # модели репозитория
        module.go       # описание зависимостей для dependency injection
```

Запрос в микросервис должен проходить следующий путь:
`хендлер` <-> `сервис` <-> `репозиторий`.

- Для http запросов это:
  - `http` <-> `service` -> `storage`
- Для консумеров/миграций это:
  - `cmd/consumer` <-> `service` <-> `storage`

Подход предполагает запуск приложения через точку входа `cmd/app`, запуск консумеров через `cmd/consumer`
и миграций через `cmd/migration`. При старте приложения в kubernetes задается количество подов (экземпляров приложения),
для каждой точки входа, для основного приложения - 3, для консумера например кафки - 2, для миграций - 1.

### Запуск приложения

1. Склонируйте репозиторий `git clone https://github.com/malekhin/demo.git` в рабочую директорию;
2. Скопируйте `config.dev.yaml.dist` в `config.dev.yaml`
3. Соберите приложение `make build`, приложение запуститься автоматически;
   - В дальнейшем используйте `make up` и `make down` для старта и остановки приложения;
3. Проверьте что сваггер открывается http://localhost:8961/swagger/index.html#

### Команды
| Команда                 | Описание                                     |
|-------------------------|----------------------------------------------|
| make build              | Сборка приложения                            |
| make up                 | Запуск приложения                            |
| make down               | Остановка приложения                         |
| make restart            | Рестарт приложения                           |
| make swag               | Генерация swagger                            |
| make generate           | Генерация моков                              |
| make lint               | Запуск проверки линтером                     |
| make test               | Запуск тестов                                |
| make test-coverage      | Запуск тестов с проверкой процентов покрытия |
| make test-visualization | Создать HTML-файл с визуализацией тестов     |
| make shell              | Зайти в контейнер приложения                 |
| make redis              | Запустить redis-cli к контейнере редиса      |
| make db                 | Запустить psql к контейнере постргеса        |
| make logs               | Посмотреть логи приложения                   |
| make drop               | Очистить БД                                  |
| make migration          | Создать новую миграцию                       |

### Запуск и тесты в локальном окружении
1. Скопируйте `config.dev.yaml.dist` в `config.local.yaml`
2. Заменить в конфиге порты postgres и mysql на 15506 и 13306 соответственно 
3. Внутри среды разработки во время создания конфигурации запуска в поле ENRIRONMENT 
укажите `ENVIRONMENT=local`
4. Теперь можно запускать приложение прямо в среде разработки, 
сваггер будет доступен по адресу http://localhost:8000/swagger/index.html#
