# Доменная зона бизнес логики

Бизнес логика делиться на доменные зоны _(нажмите для перехода в соответствующему каталогу)_:

- [`sk`](sk/) # отвечает за СК (страховые компании)
    - [`http`](sk/http/) # транспортный слой HTTP rest api
      - [`admin`](sk/http/) # хендлеры доступные только внутри микросервиса (админка сервиса)
      - [`public`](sk/http/) # хендлеры доступные в интернете
      - `api` # хендлеры доступные только внутри кластера
  - [`model`](sk/model) # модели доменной зоны
  - [`service`](sk/service) # бизнес логика
  - [`storage`](sk/storage) # репозитории (доступ к БД, другие..)
- `history` # отвечает за работу с историей

Смысл доменных зон изолировать логику приложения друг от друга.
Помимо того что это один из принципов SOLID, 
на практике изоляция дает гибкость и простоту в изменении кода,
то-есть если необходимо изменить логику которая относится к СК, программист изменяет только её,
если необходимо изменить только логику логирования истории, то, соответственно меняется только она.
При это доменная зона СК не знает ничего о внутренней структуре зоны истории,
достигается это применением интерфейсов и автомонтирования в [module.go](module.go).
